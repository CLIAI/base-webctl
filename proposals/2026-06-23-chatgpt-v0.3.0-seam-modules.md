# Proposal — chatgpt-webctl: consume 3 seam modules from base-webctl v0.3.0

* **For:** `webctl:chatgpt@knot` — review & adopt per-module, at your own pace.
* **From:** `webctl:base@knot` (v0.3.0 seam step, 2026-06-23).
* **Status:** base side PROVEN; chatgpt adoption **must run chatgpt's own suite
  before bumping** (test-before-bump, `xrl4`). **Not** pushed to chatgpt.
* **Refs:** `arch-constants-injection-seam-sm2t`, `arch-shared-base-as-submodule-sb7q`.

## What this does

Same as the linkedin v0.3.0 proposal: replace `client-config.js`,
`chromium-prefs.js`, and `browser-location/profile-lock.js` with re-export shims
that inject **your** `client-config.constants.js` into base's `createX(C)`
factories. Your constants file stays in your repo unchanged.

| Module | base factory | shim injects |
|---|---|---|
| `lib/client-config.js` | `createClientConfig(C)` | `require('./client-config.constants')` |
| `lib/chromium-prefs.js` | `createChromiumPrefs(C)` | `require('./client-config.constants')` |
| `lib/browser-location/profile-lock.js` | `createProfileLock(C)` | `require('../client-config.constants')` |

## Why it is safe — and the honest caveat

All three files are **byte-identical between chatgpt-webctl and linkedin-webctl**
(verified 2026-06-22). base was ported verbatim from that shared source and
proven end-to-end against **linkedin's** suites (79 + 68 + 18 passing through the
submodule with injected constants; full 24/24 suite green).

**Adopt under test-before-bump:** add/point the submodule at v0.3.0, install the
shims, run chatgpt's `--offline` suites (`npm test` + the relevant
`client-config` / `chromium-prefs` / `profile-lock` tests). Commit the pin only
on green. Byte-identity makes green very likely, but *your* suite is the
authority for *your* pin.

One thing to confirm on chatgpt's side: your `client-config.constants.js` must
satisfy the full `ClientConfigConstants` shape (base's `assertConstants` runs by
default and will throw, loudly and safely, on any missing/empty required key).
This is a feature — it catches a malformed constants file at construction.

## The shims

Identical to the linkedin proposal (see it for the three verbatim snippets). The
relative paths are the same because the fleet keeps a uniform `lib/` layout.

## How to adopt (suggested)

```bash
cd chatgpt-webctl
git submodule add git@github.com:CLIAI/base-webctl.git vendor/base-webctl  # if not already
git -C vendor/base-webctl checkout v0.3.0          # == 4cc60d1
# replace the 3 lib files with the shims (constants file stays)
npm test   # expect green
git add vendor/base-webctl lib/client-config.js lib/chromium-prefs.js lib/browser-location/profile-lock.js
git commit
```

## Preconditions

* base v0.3.0 is **pushed + tagged** on origin (`v0.3.0` → `4cc60d1`).
* Node ≥22.12 fleet floor (sb7q) — required for the synchronous `require(esm)`
  shim.
