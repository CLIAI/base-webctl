## What this does

chatgpt's `mounts.js` is the **canonical** body base was ported from (it carries
the WAY-1 attach-upload mount). Replace it with a re-export shim onto base's
`createMounts(C)`. Your `client-config.constants.js` stays unchanged.

| Module | base factory | shim injects |
|---|---|---|
| `lib/browser-location/mounts.js` | `mounts.createMounts(C, opts)` | `require('../client-config.constants')` + `opts.dockerfilesDir` |

## The shim

```js
'use strict';
// lib/browser-location/mounts.js ‚Äî re-export shim onto base createMounts(C).
const path = require('path');
const { mounts } = require('../../vendor/base-webctl/lib/index.js'); // require(esm)
module.exports = mounts.createMounts(require('../client-config.constants'), {
  dockerfilesDir: path.resolve(__dirname, '..', '..', 'dockerfiles'),
});
```

Note: the shim **drops** the old `const fileStaging = require('./file-staging')`.
base owns the container path now (see below), and nothing else in mounts needed
file-staging. Your `lib/browser-location/file-staging.js` stays exactly as-is ‚Äî
it is required directly by the driver, `composer.js`, `tabs.js`, and its tests;
none of those go through mounts.

## The upload mount: now base-owned, behaviour identical

* base owns `CONTAINER_UPLOAD_DIR = '/cliai-uploads'` (a fixed cross-tool container
  path ‚Äî exported from base `mounts`). The dedicated upload mount is added when
  `cfg.uploadHostPath` is set, **read-only** ‚Äî identical to your current code.
* **You keep setting `cfg.uploadHostPath`** from your host-side staging: the driver
  (`chromium-docker-xpra.js`) passes `uploadHostPath: uploadDirs.uploadHostDir`
  from `file-staging.resolveUploadDirs`. Unchanged.
* Your `file-staging.js` still defines its own `CONTAINER_UPLOAD_DIR = '/cliai-uploads'`
  for the host-side staging-dir resolver. That literal now also lives in base
  (same value). **Optional polish** (not required for adoption): have your
  `file-staging.js` import base's `mounts.CONTAINER_UPLOAD_DIR` to de-duplicate the
  constant; until then they are two identical literals ‚Äî harmless.

## The new `dockerfilesDir` seam ‚Äî read this

The v0.4.0 plan did not flag it, but `dockerfilesDir()`/`dockerfilePath()`
resolved **module-relative** (`../../dockerfiles`) in the old mounts.js. Vendored,
that points into base (no dockerfiles). base now takes `opts.dockerfilesDir`
(absolute path or thunk); the shim supplies **your** repo's `dockerfiles/` via its
own `__dirname`. A module-relative fallback exists for standalone use, but the
**vendored shim MUST inject** ‚Äî your driver + `docker-cmd.js` call
`mounts.dockerfilesDir()` / `mounts.dockerfilePath()` at build time and need the
real path.
