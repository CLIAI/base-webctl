* **From:** `webctl:base@knot` (v0.3.0 seam step, 2026-06-23).
* **Status:** PROVEN in a scratch clone; **not** pushed to linkedin.
* **Refs:** `arch-constants-injection-seam-sm2t` (Greg-approved Option A),
  `arch-shared-base-as-submodule-sb7q`, `test-cross-repo-consumer-loop-xrl4`.
* **Builds on:** the docker-ctl pilot + the v0.2.0 leaf modules.

## What this does

Replaces three **config-bound** modules with re-export shims that inject the
consumer's own per-repo constants into base's `createX(C)` factories (the seam).
base never imports your values; you pass them in. Each module is independently
adoptable.

| Module (consumer path) | base factory | shim |
|---|---|---|
| `lib/client-config.js` | `createClientConfig(C)` | injects `require('./client-config.constants')` |
| `lib/chromium-prefs.js` | `createChromiumPrefs(C)` | injects `require('./client-config.constants')` |
| `lib/browser-location/profile-lock.js` | `createProfileLock(C)` | injects `require('../client-config.constants')` |

**`lib/client-config.constants.js` STAYS in your repo, unchanged** — it is the
per-repo seam (Greg-approved 2026-05-30). Only the *resolver/behaviour* moves to
base.

**Net diff across the three:** `+10 / -1241` lines.

The factories keep base **top-level-await-free**, so the synchronous

runs by default inside each factory — a malformed constants object fails fast and
loud (pass `{ assert: false }` only as an advanced escape hatch).

## The shims (verbatim)

```js
// lib/client-config.js
'use strict';
const C = require('./client-config.constants');
const { createClientConfig } = require('../vendor/base-webctl/lib/client-config.js');
module.exports = createClientConfig(C);
```
```js
// lib/chromium-prefs.js
'use strict';
const C = require('./client-config.constants');
const { createChromiumPrefs } = require('../vendor/base-webctl/lib/chromium-prefs.js');
module.exports = createChromiumPrefs(C);
```

```js
'use strict';
const C = require('../client-config.constants');

module.exports = createProfileLock(C);
```

## Note on the shared/per-repo split

base now **owns the values that were identical** across the tools (the port
offsets, `ENV_SUFFIXES`, `CACHE_BASE/CONFIG_BASE`, `LEGACY_PORTHOST_PREFIX`,
`DEFAULT_HOST='127.0.0.1'`). Your `client-config.constants.js` keeps only the
**genuinely per-repo** subset. No action needed — the resolver reads the shared
ones from base and the per-repo ones from your injected `C`; behaviour is
identical (proven by the 79/68/18 suites).

## How to adopt (suggested)

```bash

```
