## What this does

Replace `lib/browser-location/mounts.js` with a re-export shim that injects
**your** `client-config.constants.js` into base's `createMounts(C)` factory.
Your constants file stays in your repo unchanged.

| Module | base factory | shim injects |
|---|---|---|
| `lib/browser-location/mounts.js` | `mounts.createMounts(C, opts)` | `require('../client-config.constants')` + `opts.dockerfilesDir` |

## The shim

```js
'use strict';
// lib/browser-location/mounts.js — re-export shim onto base createMounts(C).
const path = require('path');
const { mounts } = require('../../vendor/base-webctl/lib/index.js'); // require(esm)
module.exports = mounts.createMounts(require('../client-config.constants'), {
  // Dockerfiles are CONSUMER-owned (they live in THIS repo, not in base), so the
  // dir must be injected — base ships no dockerfiles/.
  dockerfilesDir: path.resolve(__dirname, '..', '..', 'dockerfiles'),
});
```

## Why it is safe

* base's `mounts.js` was ported verbatim from the chatgpt canonical (which is the
  pre-upload linkedin body **plus** the upload gate) and is file-staging-agnostic:
  the dedicated `/cliai-uploads` mount is added **only** when `cfg.uploadHostPath`
  is set. **linkedin never sets `uploadHostPath`** → mount list is the same two
  entries (X11 socket + profile) it has today. Behaviour unchanged.
* Proven 2026-06-23 against linkedin @ HEAD via a vendored worktree + this exact
  shim: `test/docker-mode-test.js` **57/0**, `test/browser-location-test.js`
  **23/0** (the browser-location orchestrator + driver both `require('./mounts')`,
  so the drop-in is proven through downstream callers, not just the mount tests).

## The new `dockerfilesDir` seam — read this

The v0.4.0 plan did not flag it, but `dockerfilesDir()`/`dockerfilePath()` in the
old mounts.js resolved **module-relative** (`../../dockerfiles`). Vendored at
`vendor/base-webctl/lib/browser-location/mounts.js`, that would resolve into base
(which has no dockerfiles). So base takes the dir as an injected `opts.dockerfilesDir`
(absolute path or thunk); the shim above supplies **your** repo's `dockerfiles/`
via its own `__dirname`. There is a module-relative fallback for standalone use,
but **the vendored shim MUST inject** — do not rely on the fallback.
