## What this does

Replace the **five** remaining `browser-location/` modules with re-export shims
onto base's factories. Your `client-config.constants.js` stays unchanged. This
completes the `browser-location/` migration (mounts shipped in v0.4.0).

| Module | base factory | shim injects |
|---|---|---|
| `browser-location/mounts.js` | `mounts.createMounts(C, o)` | constants + `dockerfilesDir` |
| `browser-location/chromium-docker-xpra.js` | `chromiumDockerXpra.createChromiumDockerXpra(C, o)` | constants + `dockerfilesDir` + `version` |
| `browser-location/registry.js` | `registry.createRegistry(C, o)` | constants + `dockerfilesDir` + `version` |
| `browser-location/xpra-attach.js` | `xpraAttach.createXpraAttach(C)` | constants |
| `browser-location/index.js` | `browserLocation.createBrowserLocation(C, o)` | constants + `dockerfilesDir` + `version` |

**linkedin injects NO `fileStaging`** — it has no `file-staging.js`, so the driver
runs its byte-unchanged no-upload path (no `/cliai-uploads` mount, no upload env
var, no upload dirs in `inspect()`). The driver, registry, and index forward the
same opts down the chain.

## The shims

```js
// browser-location/mounts.js
'use strict';
const path = require('path');
const { mounts } = require('../../vendor/base-webctl/lib/index.js');
module.exports = mounts.createMounts(require('../client-config.constants'), {
  dockerfilesDir: path.resolve(__dirname, '..', '..', 'dockerfiles'),
});
```

## xpra-attach — you are the CANONICAL side (a no-op adoption for you)

base shipped **your** xpra-attach body (it parameterizes `opts.html5Port`, default
`port+1`; chatgpt had hardcoded 14501). So adopting base's `createXpraAttach(C)`
changes nothing for linkedin — same behaviour, now shared.

## The `version` + `dockerfilesDir` seams — read this

Two values in the old driver/mounts were **repo-relative** and break once base is
vendored: `dockerfilesDir()` (`../../dockerfiles`) and the lock's
`require('../../package.json').version`. base now takes both as injected opts; the
shims supply them from **your** repo via `__dirname` / your `package.json`. Don't
rely on the fallbacks.
